| 风险 | 影响 | 应对策略 |
|---|---|---|
| 迁移体系引入晚于业务开发 | 多人并行时表结构冲突 | Week 1 前两天优先落地 Flyway |
| 用户端 UI 频繁调整影响联调 | 接口联调窗口被挤压 | UI 改造按“先行为一致、后视觉细节”推进 |

## 6. V3+ 迭代计划（版本化）

### V3（分类信息增强版，建议 1.5~2 周）

目标：把 V2 从“可用”升级到“可运营”。

- 分类信息能力：
  - 扩展分类：宠物走失、房屋租售、本地服务、拼车搭子
  - 详情增强：作者信息、发布时间、状态时间线
  - 我的发布：按状态分组（待审/上架/下架）
  - 后台增强：批量审核、批量下架、审核备注模板联动
- 验收：
  - 发布 -> 审核 -> 上架 -> 下架 -> 恢复 全链路可追踪
  - 后台可按分类/状态/社区筛选并批量处理

### V3.5（运营与风控增强，建议 1 周）

目标：提升运营效率和治理透明度。

- 模板与审计：
  - 备注模板支持版本/启停历史
  - 操作日志支持筛选导出（模板、分类、活动三模块统一）
- 风控联动：
  - 举报处理动作可联动分类信息状态
  - 高频违规用户进入观察名单
- 验收：
  - 审计日志完整覆盖关键动作
  - 回溯到“谁、在何时、因何原因、处理了哪个对象”

### V4（通知中心与消息闭环，建议 1~1.5 周）

目标：把“状态变化”可感知化，提升用户反馈闭环。

- 通知事件：
  - 分类信息审核通过/驳回通知
  - 活动报名通知完善（已读/未读、批量已读）
  - 系统公告通知
- 用户端：
  - 通知中心按类型分栏、支持跳转到详情页
- 验收：
  - 核心运营动作均能触发用户可见通知
  - 未读数、已读状态与详情跳转一致

### V4.5（稳定性与上线工程化，建议 1 周）

目标：进入“可持续发布”节奏。

- 工程化：
  - 回归脚本分层：`smoke` / `regression`
  - 最小集成测试：认证、分类审核流、活动处理流、模板权限流
  - 发布与回滚手册演练
- 验收：
  - 新版本发布前 30 分钟内完成全链路验证
  - 关键问题能在一版内回滚与复现
